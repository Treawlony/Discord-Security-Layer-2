# Performance Review: v0.4.0 — Operational Improvements & Admin Tooling

## Scope

Reviewing all five features for N+1 queries, missing indexes, unnecessary computation, and bundle/memory impact.

---

## Feature 1 — Graceful Shutdown

**DB impact:** `db.$disconnect()` is a single connection-pool teardown call. It waits for in-flight queries to complete, then closes. No performance concern.

**CPU/memory:** The `isShuttingDown` boolean guard is a single flag read/write — negligible.

**Assessment:** No performance concerns.

---

## Feature 2 — Rate Limiting on /elevate

### In-memory Map operations

Each `/elevate` invocation performs:
1. `Map.get(key)` — O(1)
2. `Array.filter()` — O(n) where n = timestamps in the window, bounded by `RATE_LIMIT_MAX_ATTEMPTS = 3`
3. `Array.push()` — O(1) amortised
4. `Map.set(key)` — O(1)

Total: effectively O(1). No DB round-trip. No bcrypt computation on rate-limited requests. This is a strict improvement over the previous behaviour where every spam invocation would reach bcrypt.

### Memory

Each Map entry: one string key (~30 chars) + array of at most 3 timestamps (8 bytes each). Under 100 bytes per active user. At 10,000 concurrent unique guild+user pairs: ~1 MB. Negligible.

Stale entries are not actively evicted — they persist as empty or small arrays until the user next uses `/elevate`. At 100,000 historical unique pairs with zero active entries, this would be ~5 MB. Still negligible for a single-instance Node.js process.

### Net effect

Rate-limited requests now exit in microseconds (Map lookup + editReply) rather than triggering a full DB + bcrypt pipeline. Spam protection also prevents unnecessary DB write pressure on `pimUser.update` (failed-attempt counter increments).

**Assessment:** Strictly positive performance impact. No concerns.

## Feature 3 — Bulk Eligibility Assignment

### Query analysis

For each role (up to 3 unique roles), `processRole()` executes:
1. `db.eligibleRole.findUnique({ where: { pimUserId_roleId: ... } })` — indexed lookup on the `@@unique([pimUserId, roleId])` compound key. O(log n). Fast.
2. `db.eligibleRole.upsert(...)` — same compound key. O(log n).
3. `db.auditLog.create(...)` — indexed insert. Fast.

Plus one pre-loop:
- `db.pimUser.findUnique({ where: { discordUserId_guildId: ... } })` — indexed on `@@unique([discordUserId, guildId])`.
- `client.guilds.fetch(guildId)` + `guild.members.fetchMe()` — single Discord API call.

**Total DB round-trips:** 1 (pimUser) + 3 × (1 findUnique + 1 upsert + up to 1 auditLog create) = worst case 10 sequential DB calls.

**N+1 risk:** The loop over roles is sequential (`for...of` with `await` inside `processRole`). For 3 roles this means 3 sequential processRole calls, each containing up to 3 DB calls. This is at most 9–10 DB calls in sequence.

**Assessment:** For the bounded maximum of 3 roles, sequential processing is acceptable. The latency is dominated by Discord's 3-second acknowledgement window; 10 small indexed DB calls complete in well under 100ms on a local-network PostgreSQL connection. Not worth parallelising — the added complexity is not justified for 3 items.

**No N+1 concern** — the loop iterates over a bounded constant (max 3 items), not an unbounded result set.

## Feature 4 — /watchtower-list Completeness

### Additional query

The command now executes two `findMany` queries instead of one:

| Query | Index used | Expected rows |
|---|---|---|
| `eligibleRole.findMany({ where: { guildId } })` | guildId column (table scan on eligible_roles filtered by guildId) | Low (admins review periodically) |
| `activeElevation.findMany({ where: { guildId } })` | guildId column | Very low (active sessions are sparse) |

Neither table has an explicit `@@index([guildId])` in `schema.prisma`. However:
- `EligibleRole` has `@@unique([pimUserId, roleId])` and `guildId` as a plain column
- `ActiveElevation` has `@@unique([pimUserId, roleId])` and `guildId` as a plain column

For typical guild sizes (tens to hundreds of rows), a guildId filter on a small table is fast regardless of index presence. However, at scale (thousands of eligibility records per guild, many guilds), a missing index on `guildId` would cause a full table scan.

**RECOMMENDATION (non-blocking):** Consider adding `@@index([guildId])` to both `EligibleRole` and `ActiveElevation` in a future migration. This is not urgent for the current deployment scale but is good hygiene. The `AuditLog` model already has `@@index([guildId, createdAt])` — the same pattern should be applied to other models.

**For this release:** The existing query pattern is unchanged from the pre-existing list command (which already uses `guildId` filtering without an explicit index). No regression introduced.

**Assessment:** One additional `findMany` query added, both queries are bounded in result size by Discord embed field limits (20/5 respectively). No performance concern for current deployment scale.

## Feature 5 — /watchtower-audit Command

### Query analysis

| Subcommand | Query | Index |
|---|---|---|
| `user` | `auditLog.findMany({ where: { guildId, discordUserId }, orderBy: { createdAt: "desc" }, take: N })` | `@@index([discordUserId])` — exists. `@@index([guildId, createdAt])` — exists. |
| `recent` | `auditLog.findMany({ where: { guildId }, orderBy: { createdAt: "desc" }, take: N })` | `@@index([guildId, createdAt])` — exists. |

Both queries hit existing indexes. The `take: N` (max 25) bounds the result set to a small constant. PostgreSQL will use the index for the ORDER BY + LIMIT pattern efficiently (index scan + early termination).

**For the `user` subcommand:** The query filters by both `guildId` and `discordUserId`. The `@@index([discordUserId])` index covers `discordUserId` but not the compound `(guildId, discordUserId)`. PostgreSQL may use the `discordUserId` index and then filter on `guildId` in memory, or it may use the `(guildId, createdAt)` index and filter on `discordUserId`. With `take: 25` the result set is tiny either way.

**RECOMMENDATION (non-blocking):** A compound `@@index([guildId, discordUserId, createdAt])` on `AuditLog` would be the most efficient index for the `user` subcommand at scale. This is a schema change (new migration) and is deferred to a future minor version.

### Embed construction

`buildAuditEmbed()` iterates over at most 25 entries and performs string concatenation. No DB calls inside the loop. The character-budget check (`charCount > EMBED_CHAR_BUDGET`) terminates early on oversized content. Total CPU time: microseconds.

**Assessment:** No performance concerns for current deployment scale. Both queries use existing indexes and are bounded by `take: 25`.
