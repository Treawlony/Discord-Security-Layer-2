## Must Fix

### MF-1 — `audit.ts` missing from `ADMIN_COMMANDS` in admin-guard.test.ts

**File:** `tests/admin-guard.test.ts`
**Status:** RESOLVED during this review

The canonical admin-guard loop iterates `ADMIN_COMMANDS` and asserts five structural properties on every admin command file (admin import, `getOrCreateGuildConfig` call, `isWatchtowerAdmin` call, permission-denied reply text, `GuildMember` import). The new `audit.ts` command was not in that list, meaning a future accidental removal of the guard would not be caught by the existing loop.

`"audit"` has been added to `ADMIN_COMMANDS`. All five guard assertions now pass for `audit.ts` (55/55 in admin-guard.test.ts). The `AUDIT_COMMANDS` list (which checks for `isWatchtowerAdmin: true` in audit log metadata) correctly omits `audit` since the command is read-only and does not write an audit log entry.

## Should Fix

### SF-1 — `elevate.ts`: `elevateCooldowns` Map grows without a hard upper bound

**File:** `src/commands/user/elevate.ts`, line 27
**Status:** RESOLVED

The `isRateLimited` function now relies on the natural replacement behaviour of the allowed path: after pruning stale timestamps, if `recent` was all-stale (becomes `[]`), the subsequent `recent.push(now); elevateCooldowns.set(key, recent)` replaces the old stale array with a single-element fresh array. This keeps the Map bounded to only users who have made a recent attempt, without requiring an explicit `delete` call.

## Positive Observations

These are things the implementation does particularly well, worth calling out explicitly:

1. **Rate-limit key design** (`src/commands/user/elevate.ts`): Keying the cooldown map on `${guildId}:${userId}` rather than `${userId}` alone is exactly right for a multi-guild bot. Many implementations get this wrong and create cross-guild interference.

2. **`isRateLimited` as a pure side-effect-free helper**: The function handles both the read and write to the Map cleanly in one pass. The "update map with pruned list even on rejection" behaviour prevents the Map from silently re-inflating with stale entries on a rejected call.

3. **`processRole` return-type union** (`src/commands/admin/assign.ts`): The discriminated union `RoleOutcome` with exhaustive `switch` in `outcomeLabel` means TypeScript will flag any future status variant that isn't handled. This is the correct pattern for extensible multi-status reporting.

4. **`findUnique` before `upsert` for idempotency detection** (`src/commands/admin/assign.ts`): Using a pre-check `findUnique` rather than relying on timestamp comparison of `createdAt`/`updatedAt` (which has millisecond precision issues) is the reliable approach. The extra round-trip is justified.

5. **`EMBED_CHAR_BUDGET` and the early-break loop** (`src/commands/admin/audit.ts`): The budget check accounts for the title length as a seed and breaks before adding a field that would overflow. This is defensive and correct — many embed-builder implementations only discover the limit when Discord rejects the payload.

6. **`isShuttingDown` guard** (`src/index.ts`): Double-signal protection is frequently overlooked. Registering both SIGTERM and SIGINT without the guard would cause `db.$disconnect()` to be called twice and `process.exit` to race.

7. **`eventTypeEmoji` export** (`src/lib/audit.ts`): Promoting the function to a named export rather than duplicating the emoji map in the new `audit.ts` command is the right call. One source of truth; testable in isolation.
