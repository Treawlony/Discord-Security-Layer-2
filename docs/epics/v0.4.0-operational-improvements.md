# Epic: v0.4.0 — Operational Improvements & Admin Tooling

## Business Value

Discord Watchtower v0.3.0 has proven the core PIM flow. This epic addresses three operational gaps that reduce trust in production deployments and one admin tooling gap that limits auditability:

1. **Graceful shutdown** — Portainer stack redeploys currently send SIGTERM with no handler, risking mid-write DB corruption and orphaned Prisma connections. Zero dropped writes on redeploy is a hard requirement for a security-sensitive tool.
2. **Rate limiting on /elevate** — The existing lockout mechanism handles brute-force, but a user (or script) can spam the command faster than bcrypt can respond, creating unnecessary load and noisy audit logs. An in-memory per-user cooldown provides a first-line spam gate.
3. **Bulk eligibility assignment** — Admins frequently need to assign 2–3 roles to a user in one operation (e.g., on-boarding a new privileged user). The current one-role-per-command UX means 3 commands, 3 audit log entries, 3 confirmation messages. Bulk assign reduces admin friction.
4. **`/watchtower-list` completeness** — The command currently shows only eligible role assignments but omits currently active elevations. Admins cannot see at a glance who is elevated right now and when their session expires. This is a critical operational visibility gap.
5. **`/watchtower-audit` command** — There is no way to query the audit log from Discord. Admins must access the database directly to investigate an incident. A Discord-native audit query command closes this gap.

## Success Metrics

- Graceful shutdown: `npm run build && docker compose restart bot` completes with no Prisma "Client already disconnected" errors in logs.
- Rate limiting: Spamming `/elevate` 5 times in 10 seconds produces a "Please slow down" ephemeral after the 4th attempt (within the 3-per-60s window) with no additional audit log entries generated.
- Bulk assign: A single `/watchtower-assign user:@u role1:@A role2:@B role3:@C` produces one command interaction, one combined reply, and three `ELIGIBILITY_GRANTED` audit events.
- `/watchtower-list`: The embed includes both an "Eligible Roles" section and an "Active Elevations" section with Discord relative-timestamp expiry.
- `/watchtower-audit`: `/watchtower-audit user:@u` returns up to 10 entries for that user; `/watchtower-audit recent:20` returns up to 20 guild-wide entries; both fit within Discord's 2000-char embed limit.

## Scope Boundaries

### In scope
- `src/index.ts` SIGTERM/SIGINT handler with cron stop + Prisma disconnect
- `src/commands/user/elevate.ts` in-memory per-user cooldown map (not persisted)
- `src/commands/admin/assign.ts` extended to role1/role2/role3 options
- `src/commands/admin/list.ts` extended to include active elevations section
- `src/commands/admin/audit.ts` new command (user subcommand + recent subcommand)
- `src/commands/user/help.ts` updated to document `/watchtower-audit`
- No schema changes required (audit log, active elevations, eligible roles already exist)

### Out of scope
- Persistent rate limit storage (Redis or DB) — in-memory is sufficient for single-instance deployment
- Pagination UI with Discord buttons for audit results — truncation to 2000 chars is sufficient for v0.4.0
- Audit log export to CSV/file
- Session extension via `/watchtower-list` (separate UX feature)

## Priority

**High** — Graceful shutdown and rate limiting are operational reliability items. Bulk assign, list completeness, and audit command are quality-of-life items that reduce admin toil for current production users.

## Version Target

v0.4.0 (MINOR — new features, new command, no schema changes)
