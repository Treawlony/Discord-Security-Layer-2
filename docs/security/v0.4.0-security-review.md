## OWASP Top 10 Assessment

### A01 — Broken Access Control

**Feature 1 (Shutdown):** No Discord-facing surface. No access control concern.

**Feature 2 (Rate Limiting):**
- FINDING: PASS. Rate limiting is a spam gate, not a security boundary. The existing bcrypt lockout mechanism (which is the actual brute-force control) is unchanged and fires correctly after the rate limit. The rate limit message does not reveal the window size or attempt threshold, preventing adversarial tuning.
- The `isRateLimited` check occurs before any DB access, meaning a rate-limited attacker cannot force bcrypt computations regardless of how fast they fire.

**Feature 3 (Bulk Assign):**
- FINDING: PASS. `isWatchtowerAdmin()` is called as the first action after `deferReply`, before any of the role processing loop runs. An unauthorized caller receives the permission-denied reply and execution stops.
- The admin-role guard inside `processRole()` prevents the Watchtower Admin role itself from being made eligible — this guard was present in the single-role version and is preserved correctly.
- The hierarchy check inside `processRole()` prevents assignment of roles the bot cannot manage, consistent with the single-role version.
- All DB operations are scoped by `guildId` (passed explicitly into `processRole`).

**Feature 4 (List):**
- FINDING: PASS. `isWatchtowerAdmin()` called immediately after `deferReply`. Both queries (`eligibleRole.findMany` and `activeElevation.findMany`) are scoped by `guildId`. No cross-guild data leakage is possible.

**Feature 5 (Audit Command):**
- FINDING: PASS. `isWatchtowerAdmin()` called immediately after `deferReply`. Both queries are scoped by `guildId`. The `user` subcommand additionally filters by `discordUserId`. No entry from another guild can appear in results.
- FINDING: PASS. The `metadata` JSON field from audit log entries is intentionally not rendered in the embed. This is correct — metadata can contain admin user IDs and other internal context that should not be surfaced in a Discord channel embed visible to anyone who can see the channel.

### A02 — Cryptographic Failures

No cryptographic operations in any of the five features. Passwords are handled exclusively in the pre-existing `verifyPassword` (bcrypt) path. No new password handling introduced.

### A03 — Injection

**SQL/DB injection:** All DB queries use Prisma's typed query builder. No raw SQL anywhere in the changed files. No injection surface introduced.

**Discord message injection:** All user-supplied data rendered in Discord messages is either:
- A Discord mention (`<@userId>`) — Discord sanitises these and they cannot inject arbitrary formatting
- A role name fetched from Discord's API (trusted source, not user-typed input)
- A timestamp rendered as `<t:unix:R>` (numeric, cannot inject)
- A Discord username from `target.username` (displayed in embed title — Discord-sanitised)

No XSS concern in Discord's context (messages are not HTML-rendered).

### A04 — Insecure Design

**Rate limiting design:**
- The in-memory Map resets on bot restart. This is an intentional trade-off documented in the design: the rate limit is a spam gate, not a security gate. Brute-force is handled by the persistent lockout mechanism. PASS.
- The `elevateCooldowns` Map is module-level, meaning it is shared across all guild interactions on the same process instance. The key includes `guildId` so there is no cross-guild interference. PASS.
- Memory growth: the Map grows by one entry per unique `guildId:userId` pair that uses `/elevate`. With stale-entry pruning on next access, entries for inactive users become empty arrays but are not deleted. At scale (thousands of users) this is a few hundred KB at most. Acceptable for a single-instance deployment. No concern for typical Discord bot usage.

**Shutdown design:**
- The `isShuttingDown` guard prevents duplicate shutdown sequences on double-signal. PASS.
- `db.$disconnect()` is awaited before `process.exit(0)`, ensuring all in-flight Prisma operations complete. PASS.
- `client.destroy()` closes the Discord WebSocket cleanly. PASS.

### A05 — Security Misconfiguration

**Audit command — no `setDefaultMemberPermissions`:** Consistent with all other admin commands per CLAUDE.md convention. `isWatchtowerAdmin()` is the runtime gate. PASS.

**New command registration:** The new `watchtower-audit` command is auto-discovered by `commandLoader.ts` and registered globally. No special configuration needed. The admin gate is runtime-enforced. PASS.

### A06 — Vulnerable and Outdated Components

No new dependencies introduced in any of the five features. All features use existing packages (node-cron, discord.js, Prisma). PASS.

### A07 — Identification and Authentication Failures

The rate limiter keys on Discord's `user.id` and `guildId`, both of which are provided by Discord's gateway and cannot be spoofed by the end user in a slash command interaction. PASS.

The audit command does not expose any authentication data. The `metadata` field (which could contain hashed password-adjacent data or admin user IDs) is intentionally not rendered. PASS.

### A08 — Software and Data Integrity Failures

No build pipeline changes, no new supply chain dependencies. PASS.

### A09 — Security Logging and Monitoring Failures

**Audit log coverage review for new features:**

| Event | Logged? | Notes |
|---|---|---|
| Rate limit rejection | No | Intentional — logging every spam attempt would itself create log noise. The legitimate failed-attempt audit log fires on the next non-rate-limited attempt if the password is wrong. |
| Role newly assigned (bulk) | Yes — ELIGIBILITY_GRANTED per role | PASS |
| Role already assigned (bulk) | No | Correct — no state change occurred |
| Role skipped (bulk) | No | Correct — no state change occurred |
| Audit query run | No | Read-only operation; no state change. Consistent with how /watchtower-list is handled. |
| Shutdown initiated | Console log only | Appropriate — no guild context available, DB may be disconnecting |

All existing event audit log coverage is unchanged.

### A10 — Server-Side Request Forgery

No HTTP requests made in any changed file. Discord API calls go through the discord.js client library. No SSRF surface introduced. PASS.

## Additional Security Observations

### Observation 1 — Bulk Assign: No partial-failure audit gap

When `role2` is skipped due to hierarchy failure but `role1` succeeds, only `role1` gets an `ELIGIBILITY_GRANTED` audit entry. The skipped role leaves no audit trace. This is correct behaviour (no state change = no audit entry), but operators should be aware that a skipped role in the reply message is the only record of the skip attempt. No finding — by design.

### Observation 2 — Rate Limit Map: No maximum size bound

The `elevateCooldowns` Map has no explicit maximum entry count. A sufficiently large number of unique guild+user pairs could grow the Map indefinitely. In practice, entries are negligibly small (`~50 bytes` per active user) and the bot serves a bounded number of guilds. Not an actionable finding for a single-instance deployment, but worth noting for future multi-instance or very large-scale deployments.

### Observation 3 — Audit embed: username displayed from Discord API

`/watchtower-audit user:@User` uses `target.username` in the embed title. This is fetched from Discord's resolved user objects and cannot be manipulated by the invoking user. PASS.

### Observation 4 — Graceful shutdown: no timeout enforcement

The shutdown handler awaits `db.$disconnect()` indefinitely. If Prisma hangs (e.g., due to a long-running transaction), the process will not exit. For Portainer's purposes this is acceptable — Portainer sends SIGTERM and then SIGKILL after a configurable timeout (default 10s). The SIGKILL provides the backstop. No finding.
