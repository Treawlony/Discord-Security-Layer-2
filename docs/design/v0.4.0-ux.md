# UX Specification: v0.4.0 — Operational Improvements & Admin Tooling

## Overview

All five features are Discord slash-command interactions. There are no new screens in the traditional sense — the "UI" is Discord's slash command picker, ephemeral reply messages, and embeds. This document specifies the exact text, structure, and interaction flow for every user-visible output.

---

## Feature 1 — Graceful Shutdown

### User-Visible Impact
None directly. The shutdown is an operator-level concern. The only observable output is console logs.

### Console Log Specification
```
[Shutdown] Signal received: SIGTERM (or SIGINT)
[Shutdown] Stopping cron job...
[Shutdown] Cron job stopped.
[Shutdown] Disconnecting from database...
[Shutdown] Database disconnected.
[Shutdown] Destroying Discord client...
[Shutdown] Discord client destroyed.
[Shutdown] Goodbye.
```
If an error occurs during shutdown:
```
[Shutdown] Error during shutdown: <error message>
```

### Accessibility
N/A — no Discord UI surface.

## Feature 2 — Rate Limiting on /elevate

### Trigger
User runs `/elevate password:...` more than 3 times within any 60-second rolling window.

### Response
Ephemeral reply (the deferred reply is edited):
```
You are sending commands too quickly. Please wait a moment before trying again.
```

### Interaction Flow
```
User types /elevate (4th time in <60s)
  → Bot defers (ephemeral)
  → Bot checks cooldown map → limit exceeded
  → Bot edits deferred reply with rate-limit message
  → Returns immediately (no DB access)
```

### Normal flow (not rate-limited) — unchanged
No visual change to the happy path.

### Accessibility
- Message is plain text, no embeds. Screen-reader friendly.
- Message does not reveal the exact window or threshold to avoid gaming.

## Feature 3 — Bulk Eligibility Assignment

### Command Signature (Discord UI)
```
/watchtower-assign user:@User role1:@Role [role2:@Role] [role3:@Role]
```
- `role1` is required (replaces old `role`)
- `role2`, `role3` are optional

### Reply Format

#### All roles newly assigned
```
Role eligibility assigned for @User:
• **RoleName1** — assigned
• **RoleName2** — assigned
• **RoleName3** — assigned

They can use /elevate to request any of these roles.
```

#### Mixed result (new + already assigned + skipped)
```
Role eligibility processed for @User:
• **RoleName1** — assigned
• **RoleName2** — already assigned (no change)
• **RoleName3** — skipped: role is at or above the bot in the role hierarchy

Use /elevate to request any eligible role.
```

#### Single role (backward-compatible with current behaviour)
```
Role eligibility assigned for @User:
• **RoleName1** — assigned

They can use /elevate to request any of these roles.
```

#### All already assigned
```
Role eligibility processed for @User:
• **RoleName1** — already assigned (no change)
• **RoleName2** — already assigned (no change)

No changes were made.
```

#### Duplicate roles provided (role1 === role2)
Deduplicate silently before processing. Only process each unique role once.

### Accessibility
- Bullet-point list is readable by screen readers.
- Bold role names provide visual hierarchy.
- No embeds — plain ephemeral text keeps it simple.

## Feature 4 — `/watchtower-list` Completeness

### Command Signature (unchanged)
```
/watchtower-list [user:@User]
```

### Embed Structure

#### No user filter (all users)
```
Title: PIM Access Overview
Color: #5865F2 (Discord blurple)
Timestamp: now

Field: "Eligible Role Assignments" (up to 20 entries)
  Each entry:
    Name: RoleName
    Value: User: @mention | Granted by: @mention | <t:unix:D>
    Inline: false

Field: "Active Elevations" (up to 5 entries, sorted soonest-expiry first)
  Each entry:
    Name: RoleName
    Value: @mention — expires <t:unix:R>
    Inline: false

  If none: single field with value "None currently active."

Footer: "Showing X of Y assignments · Z active elevation(s)" (if truncated)
```

#### With user filter
```
Title: PIM Access — @Username
(same structure, filtered to that user)
```

#### Empty state (no assignments, no elevations)
```
Plain reply: "No PIM access records found."
(No embed — consistent with current behaviour)
```

### Design Rationale
- Eligible roles section is limited to 20 fields (down from current 25) to make room for up to 5 active elevation fields within Discord's 25-field embed cap.
- Active elevations sorted ascending by expiry because the most urgent (soonest to expire) is most operationally relevant.
- Using separate field groups (not a single merged list) keeps the two concepts visually distinct.

### Accessibility
- Relative timestamps (`<t:unix:R>`) are rendered by Discord natively and are accessible.
- All user references use `<@id>` mentions so Discord renders the username even if the admin doesn't know the ID.

## Feature 5 — `/watchtower-audit` Command

### Command Signature
```
/watchtower-audit user:@User [limit:10]
/watchtower-audit recent:10
```
Note: `user` and `recent` are subcommands, not options. Discord presents them as:
```
/watchtower-audit user   → then options: user (required), limit (optional, 1-25)
/watchtower-audit recent → then options: limit (required, 1-25, default 10)
```

### Embed Structure — user subcommand
```
Title: "Audit Log — @Username (last N)"
Color: #5865F2
Timestamp: now

One field per entry (up to N, max 25):
  Name:  "<emoji> AUDIT_EVENT_TYPE"
  Value: "<@userId> | <t:unix:R>[  | Role: RoleName]"
  Inline: false

Footer: "Showing X of Y entries" (if Y > X due to char truncation)
        "No audit log entries found." (plain reply if 0 results)
```

### Embed Structure — recent subcommand
```
Title: "Audit Log — Recent (last N)"
Color: #5865F2
Timestamp: now

(same field structure as above)
```

### Emoji Mapping (reuses existing audit.ts eventTypeEmoji logic)
```
ELEVATION_GRANTED          ⬆️
ELEVATION_EXPIRED          ⏱️
ELEVATION_REVOKED          ⬇️
ELEVATION_REQUESTED        🔑
FAILED_ATTEMPT             ⚠️
ACCOUNT_LOCKED             🔒
ACCOUNT_UNLOCKED           🔓
PASSWORD_SET               🔐
PASSWORD_CHANGED           🔄
PASSWORD_RESET             🔏
ELIGIBILITY_GRANTED        ✅
ELIGIBILITY_REVOKED        ❌
ELEVATION_EXPIRY_WARNING   ⏰
ELEVATION_EXTENDED         🔁
ELEVATION_ADMIN_REVOKED    🚫
ELEVATION_ADMIN_REVOKED_BLOCKED ⛔
ELEVATION_BLOCKED          🔴
ELEVATION_SELF_REVOKED     ↩️
(default)                  📋
```

### Truncation Behaviour
- Each embed field value is capped at 1024 chars (Discord limit).
- If adding the next entry would push total embed char count past 5500, stop and add footer: "Showing X of Y entries (truncated to fit Discord limits)".
- Metadata JSON is intentionally excluded from display (internal data, not user-readable in this context).

### Empty State
```
Plain reply: "No audit log entries found for this query."
```

### Unauthorized Access
```
Plain reply (same as other admin commands):
"You do not have permission to use this command.
A Watchtower Admin role is required. Contact your server owner to be assigned the correct role."
```

### Updated help.ts — Admin Commands section addition
```
`/watchtower-audit` — Query the audit log. Use `user:@User` to see a user's history or `recent:N` to see the latest N guild-wide events (max 25).
```
