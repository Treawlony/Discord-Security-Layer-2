# User Stories: v0.4.0 — Operational Improvements & Admin Tooling

## Story 1 — Graceful Shutdown

**As a** DevOps operator redeploying the bot via Portainer,
**I want** the bot to finish all in-flight database operations before exiting,
**So that** no audit log entries are lost and no Prisma connections are leaked during rolling restarts.

### Acceptance Criteria

1. `process.on('SIGTERM', handler)` and `process.on('SIGINT', handler)` are registered in `src/index.ts` before `client.login()`.
2. The handler stops the cron job (calls the task's `stop()` method from node-cron).
3. The handler calls `db.$disconnect()` and awaits it.
4. The handler calls `client.destroy()` to cleanly close the Discord WebSocket.
5. The handler calls `process.exit(0)` after cleanup completes.
6. The handler logs each shutdown step to `console.log` with a `[Shutdown]` prefix.
7. If any step in the handler throws, the error is logged and `process.exit(1)` is called.
8. `startExpiryJob` returns the cron task object so the shutdown handler can call `.stop()` on it.

### Data Requirements
- No new DB fields required.
- `startExpiryJob` return type changes from `void` to `ScheduledTask` (from `node-cron`).

### Edge Cases
- Double-signal (SIGTERM fired twice): second signal should be a no-op if shutdown is already in progress (use a boolean `isShuttingDown` guard).
- Cron job not yet started (startup failure before `startExpiryJob`): `stop()` call should be guarded with null check.

## Story 2 — Rate Limiting on /elevate

**As a** server security admin,
**I want** the bot to reject rapid repeated `/elevate` attempts from the same user,
**So that** users cannot spam the command to create noise in the audit log or generate unnecessary bcrypt computation.

### Acceptance Criteria

1. A module-level `Map<string, number[]>` tracks timestamps of recent `/elevate` invocations per Discord user ID (keyed as `${guildId}:${userId}` for multi-guild safety).
2. On each `/elevate` invocation, before any DB access, the bot checks if the user has made 3 or more attempts within the last 60 seconds.
3. If the rate limit is exceeded, the bot replies with an ephemeral message: "You are sending commands too quickly. Please wait a moment before trying again." and returns immediately (no DB access, no audit log entry).
4. If the rate limit is not exceeded, the invocation timestamp is appended to the user's list and stale entries (older than 60 seconds) are pruned.
5. The window is 60 seconds and the threshold is 3 attempts. These values are defined as named constants at the top of the file (not hardcoded inline).
6. The cooldown map is module-level (process lifetime); it is not persisted across restarts. This is acceptable — it is a spam gate, not a security gate.
7. The rate limit check and ephemeral reply happen before `deferReply` is called (or immediately after — see note below).

### Note on deferReply ordering
Per CLAUDE.md, `deferReply` must be the first statement. The rate-limit check should occur immediately after `deferReply` to comply with the convention while still preventing unnecessary work.

### Data Requirements
- No DB changes. The `Map` is in-process only.

### Edge Cases
- User is rate-limited and also locked out: rate limit fires first (it is checked earlier in the flow).
- Bot restarts: cooldown resets. Acceptable — prevents spam within a process lifetime.
- Multi-guild: the map key includes `guildId` so a user spamming in two guilds is tracked independently.

## Story 3 — Bulk Eligibility Assignment

**As a** Watchtower Admin onboarding a new privileged user,
**I want** to assign up to 3 roles in a single `/watchtower-assign` command,
**So that** I don't have to repeat the command multiple times for users who need access to several roles.

### Acceptance Criteria

1. `/watchtower-assign` gains two additional optional role options: `role2` and `role3` (in addition to the existing required `role` / `role1`).
2. The existing `role` option is renamed to `role1` for consistency, and remains required. (Rename in the builder; the option name in Discord will change.)
3. All provided roles are processed in a single command execution.
4. For each role, the bot performs the same hierarchy check, admin-role guard, and upsert logic as the current single-role path.
5. The reply summarises results per role: which were newly assigned vs. already assigned (idempotent detection via the upsert `update` path).
6. One `ELIGIBILITY_GRANTED` audit log entry is written per role, matching the existing per-role audit pattern.
7. If the target user has no PIM account, the command fails early (before processing any roles) with the existing "run /set-password first" message.
8. If a role fails the hierarchy check or admin-role guard, that role is skipped and reported in the reply as "skipped" with a reason; other roles continue to be processed.
9. The reply is a single ephemeral message listing all outcomes (newly assigned, already assigned, skipped).

### Process Flow

**Current state:** Admin runs `/watchtower-assign` once per role → N commands for N roles.
**Future state:** Admin runs one `/watchtower-assign` with up to 3 role options → one reply with all outcomes.

### Data Requirements
- No schema changes. Uses existing `EligibleRole` upsert.
- The `upsert` return value is used to detect new vs. existing: if `createdAt === updatedAt` within a millisecond window, it was newly created; alternatively track this via a pre-check `findUnique` before upsert.

### Edge Cases
- Only `role1` provided (role2/role3 null): behaves identically to current single-role command.
- All 3 roles already assigned: reply lists all 3 as "already assigned" (no error).
- `role1 === role2`: process the duplicate once, skip the second occurrence with a note in the reply.
- `role2` fails hierarchy, `role1` and `role3` succeed: partial success reply.

## Story 4 — `/watchtower-list` Completeness

**As a** Watchtower Admin conducting a routine access review,
**I want** `/watchtower-list` to show both eligible role assignments and currently active elevations,
**So that** I have a single command that gives me a complete picture of PIM access state.

### Acceptance Criteria

1. The command's embed is extended with a second section: "Active Elevations".
2. For each active elevation in scope (filtered by user if `user` option provided), show:
   - The user mention (`<@userId>`)
   - The role name
   - Expiry as a Discord relative timestamp (`<t:unix:R>`)
3. If there are no active elevations, the "Active Elevations" section shows "None".
4. When `user` option is provided, both sections are filtered to that user.
5. When no `user` option is provided, both sections show all entries guild-wide.
6. Active elevations are sorted by `expiresAt` ascending (soonest-to-expire first).
7. Combined field count does not exceed Discord's 25-field embed limit; if it would, eligible role fields are shown up to 20 and active elevation fields up to 5, with a footer noting truncation.
8. All DB queries are scoped by `guildId`.

### Data Requirements
- No schema changes. Uses existing `ActiveElevation` model.
- Query: `db.activeElevation.findMany({ where: { guildId, ...(targetUser && { pimUserId }) }, include: { pimUser: true }, orderBy: { expiresAt: 'asc' } })`

### Edge Cases
- No eligible roles but active elevations exist: eligible roles section shows "None assigned", elevations section is populated.
- Active elevation exists but the associated `eligibleRole` was revoked: elevation still shows (it is a live session).
- User filtered: no active elevations for that user → "None" in that section.

## Story 5 — `/watchtower-audit` Command

**As a** Watchtower Admin investigating a security incident,
**I want** to query the audit log from within Discord,
**So that** I can see recent events without needing database access.

### Acceptance Criteria

1. New command `src/commands/admin/audit.ts` with name `watchtower-audit`.
2. Two mutually exclusive subcommands (use `addSubcommand`):
   - `user` subcommand: required `user` option (User mention) + optional `limit` integer (1–25, default 10).
   - `recent` subcommand: required `limit` integer (1–25, default 10).
3. Command gates on `isWatchtowerAdmin()` immediately after `deferReply`.
4. All queries are scoped by `guildId`.
5. `user` subcommand: queries `AuditLog` where `discordUserId = target.id AND guildId = guildId`, ordered by `createdAt DESC`, limited to `limit`.
6. `recent` subcommand: queries `AuditLog` where `guildId = guildId`, ordered by `createdAt DESC`, limited to `limit`.
7. Results are rendered as a Discord embed with:
   - Title: "Audit Log — @username (last N)" or "Audit Log — Recent (last N)"
   - One field per entry: field name = `eventType` with emoji prefix, field value = `<@userId> | <t:unix:R>` plus optional role name if present.
8. If total character count of the embed would exceed 5500 chars (Discord embed limit), entries are truncated and a footer notes "Showing X of Y entries (truncated)".
9. If no entries found, reply with "No audit log entries found."
10. Command has no `setDefaultMemberPermissions` (consistent with other admin commands).
11. `help.ts` is updated to include `/watchtower-audit` in the Admin Commands section.

### Data Requirements
- No schema changes. Uses existing `AuditLog` model with existing indexes on `(guildId, createdAt)` and `(discordUserId)`.

### Edge Cases
- `limit` out of range (enforced by Discord's `setMinValue(1).setMaxValue(25)` on the option).
- User has no audit log entries in this guild: "No audit log entries found."
- Very long `metadata` JSON on an entry: field value is capped at 1024 chars (Discord field limit); metadata is not displayed (it is internal).
- Subcommand not matched (should be impossible with Discord's slash command routing, but guard with an else branch).

## Process Flow Summary

```
Current state                          Future state
─────────────────────────────────      ──────────────────────────────────────
SIGTERM → Prisma error, crash          SIGTERM → clean shutdown, graceful exit
/elevate spam → N bcrypt calls         /elevate spam → "slow down" after 3 in 60s
/assign role per cmd → N commands      /assign role1 role2 role3 → 1 command
/list → eligible roles only            /list → eligible roles + active elevations
No audit query → DB access required    /watchtower-audit → Discord-native query
```
