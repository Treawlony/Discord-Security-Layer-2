# Technical Design: v0.4.0 — Operational Improvements & Admin Tooling

## Architecture Overview

All five features are purely application-layer changes. No new database migrations are required. No new infrastructure components are introduced. The existing stack (Node.js 20, TypeScript 5, discord.js v14, Prisma v5, PostgreSQL 16, node-cron) handles everything.

---

## Feature 1 — Graceful Shutdown

### Design

**Problem:** `startExpiryJob` currently returns `void`. The cron task handle is discarded, making it impossible to call `task.stop()` from outside the function.

**Solution:** Change the return type of `startExpiryJob` to `ScheduledTask` (exported from `node-cron`) and return the task handle. The caller (`main()` in `src/index.ts`) stores the handle and passes it to the shutdown handler.

### src/jobs/expireElevations.ts — change

```typescript
// Before
export function startExpiryJob(client: Client): void {
  cron.schedule("* * * * *", async () => { ... });
}

// After
import cron, { ScheduledTask } from "node-cron";
export function startExpiryJob(client: Client): ScheduledTask {
  const task = cron.schedule("* * * * *", async () => { ... });
  return task;
}
```

### src/index.ts — additions

```typescript
// Shutdown guard
let isShuttingDown = false;

async function shutdown(signal: string, task: ScheduledTask | null): Promise<void> {
  if (isShuttingDown) return;
  isShuttingDown = true;

  console.log(`[Shutdown] Signal received: ${signal}`);

  try {
    if (task) {
      console.log("[Shutdown] Stopping cron job...");
      task.stop();
      console.log("[Shutdown] Cron job stopped.");
    }

    console.log("[Shutdown] Disconnecting from database...");
    await db.$disconnect();
    console.log("[Shutdown] Database disconnected.");

    console.log("[Shutdown] Destroying Discord client...");
    client.destroy();
    console.log("[Shutdown] Discord client destroyed.");

    console.log("[Shutdown] Goodbye.");
    process.exit(0);
  } catch (err) {
    console.error("[Shutdown] Error during shutdown:", err);
    process.exit(1);
  }
}

// In main(), after startExpiryJob:
const task = startExpiryJob(client);
process.on("SIGTERM", () => shutdown("SIGTERM", task));
process.on("SIGINT",  () => shutdown("SIGINT",  task));
```

### Type Compatibility
`node-cron` v3 exports `ScheduledTask` as a named type. `@types/node-cron` v3 (already in devDependencies) exposes it. No new type packages needed.

### Security Considerations
None. This is a process lifecycle concern only.

### Infrastructure Impact
Portainer sends SIGTERM on stack stop/redeploy. The handler fires, cleanup completes in under 5 seconds (db disconnect + WS close), then exits. No restart-policy change needed.

## Feature 2 — Rate Limiting on /elevate

### Design

**Approach:** Module-level `Map` in `src/commands/user/elevate.ts`. No external state store. Acceptable for single-instance deployment (Portainer runs one bot container).

**Key design:** Keyed by `${guildId}:${userId}` to isolate per guild. Window and threshold as named module-level constants.

```typescript
const RATE_LIMIT_WINDOW_MS = 60_000;   // 60 seconds
const RATE_LIMIT_MAX_ATTEMPTS = 3;

// Map key: `${guildId}:${userId}` → sorted array of attempt timestamps (ms)
const elevateCooldowns = new Map<string, number[]>();
```

### Rate-limit check logic (pure, no side effects)

```typescript
function isRateLimited(key: string): boolean {
  const now = Date.now();
  const attempts = (elevateCooldowns.get(key) ?? [])
    .filter(ts => now - ts < RATE_LIMIT_WINDOW_MS);

  if (attempts.length >= RATE_LIMIT_MAX_ATTEMPTS) {
    return true;
  }

  attempts.push(now);
  elevateCooldowns.set(key, attempts);
  return false;
}
```

### Placement in execute()

```typescript
export async function execute(interaction, client) {
  await interaction.deferReply({ flags: MessageFlags.Ephemeral });

  const key = `${interaction.guildId}:${interaction.user.id}`;
  if (isRateLimited(key)) {
    return interaction.editReply(
      "You are sending commands too quickly. Please wait a moment before trying again."
    );
  }

  // ... rest of existing logic unchanged
}
```

### Memory Footprint
Each entry is `[guildId:userId] → number[]` with at most `RATE_LIMIT_MAX_ATTEMPTS` timestamps. At 1000 active users, ~50 KB. Negligible.

### No Cleanup Needed
Stale entries (users who haven't used `/elevate` recently) linger harmlessly. The `.filter` in `isRateLimited` prunes old timestamps on next access. A periodic cleanup is not worth the complexity.

### Security Considerations
- This is NOT a brute-force control — the lockout mechanism handles that.
- Rate limit is bypassed on bot restart (in-memory only). Acceptable: the window is 60s and the bot restarts are infrequent.
- The error message intentionally does not reveal the exact threshold or window.
- Multi-guild isolation: the `guildId` prefix in the key prevents cross-guild interference.

## Feature 3 — Bulk Eligibility Assignment

### Command Builder Changes

```typescript
export const data = new SlashCommandBuilder()
  .setName("watchtower-assign")
  .setDescription("Assign role eligibility to a user.")
  .addUserOption(opt => opt.setName("user").setDescription("The user").setRequired(true))
  .addRoleOption(opt => opt.setName("role1").setDescription("Role to assign (required)").setRequired(true))
  .addRoleOption(opt => opt.setName("role2").setDescription("Additional role (optional)").setRequired(false))
  .addRoleOption(opt => opt.setName("role3").setDescription("Additional role (optional)").setRequired(false));
```

Note: The existing `role` option is renamed to `role1`. This is a breaking change to the command signature in Discord. Global command propagation (up to 1 hour) will update the command picker. Any user mid-interaction during the rollout window will see the old UI; their interaction will fail gracefully as discord.js will still parse the option if passed under the old name... actually Discord re-validates on the server side. The rename is safe because the command is re-registered on bot start via the `ready` event.

### Per-Role Processing Architecture

Extract the single-role processing into a local helper function to avoid code duplication:

```typescript
type RoleOutcome =
  | { status: "assigned";          roleName: string }
  | { status: "already_assigned";  roleName: string }
  | { status: "skipped";           roleName: string; reason: string };

async function processRole(
  client: Client,
  interaction: ChatInputCommandInteraction,
  guildId: string,
  config: GuildConfig,
  pimUser: PimUser,
  role: Role  // discord.js Role object
): Promise<RoleOutcome>
```

The function:
1. Checks admin-role guard (skip if `role.id === config.adminRoleId`)
2. Checks hierarchy (skip if `role.position >= botMember.roles.highest.position`)
3. Runs `db.eligibleRole.upsert()`, captures whether it was a create or update by comparing `createdAt` and `updatedAt` within 1 second (or use a pre-check `findUnique`)
4. Writes `ELIGIBILITY_GRANTED` audit log entry
5. Returns the appropriate outcome

### Idempotency Detection

Use a `findUnique` before upsert to reliably detect existing vs. new:

```typescript
const existing = await db.eligibleRole.findUnique({
  where: { pimUserId_roleId: { pimUserId: pimUser.id, roleId: role.id } },
});

await db.eligibleRole.upsert({ ... });

return existing
  ? { status: "already_assigned", roleName: role.name }
  : { status: "assigned", roleName: role.name };
```

### Deduplication

Collect the non-null role options into a `Set` keyed by role ID before processing:

```typescript
const roleOptions = [
  interaction.options.getRole("role1", true),
  interaction.options.getRole("role2"),
  interaction.options.getRole("role3"),
].filter(Boolean);

// Deduplicate by role ID
const uniqueRoles = [...new Map(roleOptions.map(r => [r!.id, r!])).values()];
```

### Hierarchy Check Fetch

The `guild.members.me` fetch is needed for the hierarchy check. Perform it once before the loop:

```typescript
const guild = await client.guilds.fetch(guildId);
const botMember = guild.members.me ?? await guild.members.fetchMe();
```

### Audit Log Behaviour

One `ELIGIBILITY_GRANTED` entry per newly assigned role. No audit entry for already-assigned or skipped roles (no state change occurred). This matches the existing pattern.

## Feature 4 — `/watchtower-list` Completeness

### Query Changes

Add a second query for active elevations alongside the existing eligible-role query:

```typescript
// Existing query (unchanged)
const assignments = await db.eligibleRole.findMany({
  where,
  include: { pimUser: true },
  orderBy: { createdAt: "desc" },
});

// New query
const elevationWhere = targetUser
  ? { guildId, pimUser: { discordUserId: targetUser.id } }
  : { guildId };

const activeElevations = await db.activeElevation.findMany({
  where: elevationWhere,
  include: { pimUser: true },
  orderBy: { expiresAt: "asc" },
});
```

### Embed Construction

Field budget: 20 slots for eligible roles, 5 slots for active elevations (total ≤ 25, Discord's embed field limit). If assignments exceed 20, the footer notes truncation. If active elevations exceed 5, they are truncated with a note.

```typescript
const embed = new EmbedBuilder()
  .setTitle(targetUser ? `PIM Access — ${targetUser.username}` : "PIM Access Overview")
  .setColor(0x5865f2)
  .setTimestamp();

// Section: Eligible Roles
const shownAssignments = assignments.slice(0, 20);
if (shownAssignments.length === 0) {
  embed.addFields({ name: "Eligible Role Assignments", value: "None assigned.", inline: false });
} else {
  for (const a of shownAssignments) {
    embed.addFields({
      name: a.roleName,
      value: `User: <@${a.pimUser.discordUserId}> | Granted by: <@${a.grantedBy}> | <t:${Math.floor(new Date(a.createdAt).getTime() / 1000)}:D>`,
      inline: false,
    });
  }
}

// Section: Active Elevations
const shownElevations = activeElevations.slice(0, 5);
if (shownElevations.length === 0) {
  embed.addFields({ name: "Active Elevations", value: "None currently active.", inline: false });
} else {
  for (const e of shownElevations) {
    embed.addFields({
      name: `[ACTIVE] ${e.roleName}`,
      value: `<@${e.pimUser.discordUserId}> — expires <t:${Math.floor(e.expiresAt.getTime() / 1000)}:R>`,
      inline: false,
    });
  }
}
```

### Footer Logic

```typescript
const footerParts: string[] = [];
if (assignments.length > 20) footerParts.push(`Showing 20 of ${assignments.length} assignments`);
if (activeElevations.length > 5) footerParts.push(`${activeElevations.length} active elevations (showing 5)`);
if (footerParts.length > 0) embed.setFooter({ text: footerParts.join(" · ") });
```

### Empty-State Handling

If both queries return zero results, return the existing plain-text reply: `"No PIM access records found."` (no embed). The embed path only runs when at least one query has results.

## Feature 5 — `/watchtower-audit` Command

### Command Builder

```typescript
export const data = new SlashCommandBuilder()
  .setName("watchtower-audit")
  .setDescription("Query the PIM audit log.")
  .addSubcommand(sub =>
    sub.setName("user")
       .setDescription("Show audit log entries for a specific user.")
       .addUserOption(opt => opt.setName("user").setDescription("Target user").setRequired(true))
       .addIntegerOption(opt =>
         opt.setName("limit").setDescription("Number of entries (1-25)").setMinValue(1).setMaxValue(25)
       )
  )
  .addSubcommand(sub =>
    sub.setName("recent")
       .setDescription("Show the most recent audit log entries for this server.")
       .addIntegerOption(opt =>
         opt.setName("limit").setDescription("Number of entries (1-25)").setMinValue(1).setMaxValue(25).setRequired(true)
       )
  );
```

### Queries

```typescript
// user subcommand
const logs = await db.auditLog.findMany({
  where: { guildId, discordUserId: target.id },
  orderBy: { createdAt: "desc" },
  take: limit,
});

// recent subcommand
const logs = await db.auditLog.findMany({
  where: { guildId },
  orderBy: { createdAt: "desc" },
  take: limit,
});
```

Both queries leverage existing indexes: `@@index([guildId, createdAt])` for the recent subcommand, and `@@index([discordUserId])` for the user subcommand (combined with the guildId filter applied in-application for the user path — acceptable given small per-guild audit log sizes).

### Embed Construction with Truncation Guard

```typescript
const EMBED_CHAR_LIMIT = 5500;

function buildAuditEmbed(title: string, logs: AuditLog[]): EmbedBuilder {
  const embed = new EmbedBuilder().setTitle(title).setColor(0x5865f2).setTimestamp();
  let charCount = title.length;
  let shown = 0;

  for (const log of logs) {
    const emoji = eventTypeEmoji(log.eventType);
    const fieldName = `${emoji} ${log.eventType}`;
    const rolePart = log.roleName ? ` | Role: **${log.roleName}**` : "";
    const fieldValue = `<@${log.discordUserId}> | <t:${Math.floor(log.createdAt.getTime() / 1000)}:R>${rolePart}`;

    if (charCount + fieldName.length + fieldValue.length > EMBED_CHAR_LIMIT) break;

    embed.addFields({ name: fieldName, value: fieldValue, inline: false });
    charCount += fieldName.length + fieldValue.length;
    shown++;
  }

  if (shown < logs.length) {
    embed.setFooter({ text: `Showing ${shown} of ${logs.length} entries (truncated to fit Discord limits)` });
  }

  return embed;
}
```

### eventTypeEmoji Reuse

The `eventTypeEmoji` function in `src/lib/audit.ts` is not exported. Rather than coupling the new command to audit.ts internals, duplicate the small mapping inline in `audit.ts` as an exported helper, or re-implement it locally in `audit.ts` as an export. The cleanest approach: export `eventTypeEmoji` from `audit.ts` and import it in `audit.ts` (it is already local there) and in the new `audit` command.

## Security Requirements

| Feature | Requirement |
|---|---|
| Graceful shutdown | No user-facing security surface |
| Rate limiting | Does not replace lockout; does not store sensitive data; key is non-sensitive guild+user ID |
| Bulk assign | Same admin gate (`isWatchtowerAdmin`) as current single-role assign; same audit trail |
| List completeness | Same admin gate; no new data exposed beyond what individual queries already expose |
| Audit command | Admin gate required; all queries scoped by `guildId`; metadata not rendered (avoids accidental sensitive data leakage) |

None. All changes are code-only. No new env vars, no new Docker services, no config changes.
