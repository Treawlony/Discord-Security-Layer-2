## Deployment Steps

### Step 1 — Pull and Redeploy in Portainer

In the Portainer stack UI for the **production** stack:

1. Click **Pull and redeploy**
2. Portainer pulls `master`, rebuilds the Docker image
3. The Dockerfile runs `prisma migrate deploy` — there are no new migrations in v1.0.0 so this is a no-op (all existing migrations are already applied)
4. The bot starts with the new code

No manual migration step is needed.

### Step 2 — Confirm Clean Startup

Check bot startup logs for the standard startup sequence with no errors:

```
[DB] Connected to PostgreSQL
[Commands] Loaded N commands
[Jobs] Elevation expiry job started
[Discord] Ready — logged in as WatchtowerBot#XXXX
```

Watch specifically for the new graceful shutdown registration line (visible in debug logging if enabled) — its absence would indicate the shutdown handler code path was not reached.

### Step 3 — Smoke Test (Manual)

Perform the following checks on the production Discord server after deployment:

**Graceful shutdown (no manual test needed)**
The shutdown behaviour is exercised automatically by Docker on every container stop/restart. Confirm the bot restarts cleanly without any "port already in use" or "database connection already closed" errors in Portainer logs during the redeploy itself.

**Rate limiting on `/elevate`**
1. Run `/elevate` (with any password) three times in rapid succession as a non-elevated user
2. The fourth attempt within 60 seconds should return: "You are sending commands too quickly. Please wait a moment before trying again."
3. Confirm no additional `FAILED_ATTEMPT` audit events are generated for the rate-limited rejections

**Bulk role assignment**
1. Run `/watchtower-assign user:@TestUser role1:@RoleA role2:@RoleB` — both roles should be assigned and reported
2. Run the same command again — both should report "already assigned (no change)" with "No changes were made." footer
3. Run with a single role as before (`role1:@RoleA` only) — should work identically to the old single-role behaviour

**Active Elevations in `/watchtower-list`**
1. While a user has an active elevation, run `/watchtower-list` — confirm an "Active Elevations" section appears in the embed with the user's name and role
2. Run `/watchtower-list user:@ElevatedUser` — confirm the filter applies to both sections
3. If no elevations are active, confirm the section shows "None currently active."

**`/watchtower-audit` command**
1. Run `/watchtower-audit recent` (no limit) — should default to 10 entries and render an embed
2. Run `/watchtower-audit recent limit:5` — should show 5 entries
3. Run `/watchtower-audit user:@SomeUser` — should show that user's audit history
4. Run as a non-admin user — should return "You do not have permission to use this command."
5. On a server with no audit log entries yet, confirm an appropriate empty-state message is shown

## Discord Command Propagation

`/watchtower-audit` is a new global slash command. After deployment:

- Global command propagation takes **up to 1 hour**
- During this window `/watchtower-audit` may not appear in the Discord UI autocomplete
- The command becomes usable immediately once propagated — no further action needed
- The `/watchtower-assign` option rename (`role` → `role1`) will also propagate within the same window

No user action is required to trigger propagation. The bot re-registers all commands on startup automatically.

## Rollback Plan

Since v1.0.0 has no database migrations, rollback is purely a code revert:

1. In Portainer, redeploy pointing to the `v0.3.0` tag or the last known-good `master` commit
2. The bot will start on the old code with the unchanged database — no data loss, no migration rollback needed
3. The `/watchtower-audit` command will disappear from Discord's command list within up to 1 hour after the old bot registers its (smaller) command set

There is no need to run any SQL to roll back v1.0.0.

## Monitoring

No new metrics or alerting required. Existing monitoring covers:

- **Graceful shutdown**: Docker/Portainer restart logs will show clean exit (exit code 0) rather than a forced kill (exit code 137) on container stop. This is the observable improvement — previously the container was killed by Docker's 10-second SIGKILL deadline.
- **Rate limiting**: Rate-limited rejections are intentionally not written to the audit log. If you observe unusual `/elevate` usage patterns, the absence of audit events is expected behaviour, not a gap.
- **`/watchtower-audit`**: The command is read-only. No new audit events are emitted when it is used.

---

## Known Limitations

- The `elevateCooldowns` rate-limit map is **in-memory only**. It resets on bot restart. A user who was rate-limited immediately before a restart can make 3 fresh attempts after restart. This is acceptable given the existing brute-force lockout mechanism provides the hard security gate.
- The rate limit is **per-process**, not distributed. If the bot were ever run as multiple replicas (not the current deployment model), each replica would have an independent cooldown map. Not a concern for the current single-instance Portainer deployment.
- `/watchtower-audit` embed truncation at 5500 characters means servers with very verbose audit entries may see fewer than the requested number of entries rendered. The footer will note when this occurs.
